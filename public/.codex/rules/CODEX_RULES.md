# twSearch 작업 순서도

이 문서는 `twSearch` 프로젝트에서 Codex가 매번 먼저 참고해야 하는 작업 기준이다.

## 1. 세션 시작 시 우선순위

새로 작업을 시작할 때는 아래 순서로 확인한다.

1. 이 문서 전체를 먼저 읽는다.
2. `git status --short`로 현재 변경 상태를 확인한다.
3. 주요 진입 파일을 확인한다.
4. 사용자 요청이 있으면 이 문서의 절차와 충돌하지 않는지 먼저 판단한다.
5. 작업 범위를 작게 나누고, 필요한 경우에만 파일을 수정한다.

## 2. 프로젝트 이해 순서

프로젝트를 분석하거나 수정할 때는 다음 순서로 본다.

1. `index.html`
2. `main.html`
3. GitHub Actions 워크플로
4. `.gitignore`
5. `.history/`는 참고만 하고, 배포나 핵심 수정에는 포함하지 않는다.

## 3. 작업 원칙

1. 사용자가 만든 변경사항은 함부로 되돌리지 않는다.
2. 삭제나 대규모 수정은 먼저 영향 범위를 확인한다.
3. 새 기능보다 기존 동작 보존을 우선한다.
4. HTML/CSS/JS가 한 파일에 섞여 있어도, 먼저 구조를 읽고 나서 수정한다.
5. 변경은 가능한 한 작은 단위로 나눈다.

## 4. 수정 순서

1. 문제를 설명할 수 있을 만큼 코드 맥락을 읽는다.
2. 수정할 파일을 정한다.
3. `apply_patch`로만 직접 편집한다.
4. 수정 후 다시 읽어서 의도대로 반영됐는지 확인한다.
5. 필요하면 간단한 검증을 수행한다.
6. 사용자에게 변경점과 남은 리스크를 짧고 분명하게 설명한다.

## 5. 이 프로젝트의 핵심 구조

1. `main.html`이 실제 앱이다.
2. `index.html`은 `main.html`로 보내는 진입용 파일이다.
3. `.github/workflows/sync-public.yml`은 공개 저장소 동기화용이다.
4. `.history/`는 작업 기록용이므로 결과물로 보지 않는다.

## 6. 검색기 로직 확인 체크리스트

1. 연도 범위 검증이 있는가.
2. 날짜 형식이 올바른가.
3. 계정 목록이 의도한 방식으로 합쳐지는가.
4. 키워드 OR 조건이 예상대로 생성되는가.
5. `filter:verified`와 `min_retweets` 조합이 맞는가.
6. 최종 검색어가 너무 길어지거나 잘못 괄호 처리되지 않는가.

## 7. 배포 확인 체크리스트

1. `main` 브랜치 기준으로 동작하는가.
2. 공개 저장소로 복사되는 파일만 정확히 포함되는가.
3. `.github/`, `.history/`, `.agents/`가 배포에서 제외되는가.
4. 비밀 값이나 개인 정보가 저장소에 남지 않는가.

## 8. 응답 기준

1. 분석 요청이면 구조와 흐름을 먼저 설명한다.
2. 수정 요청이면 먼저 실제로 변경한다.
3. 불확실한 점이 있으면 추측하지 말고 표시한다.
4. 사용자에게는 짧고 실용적으로 보고한다.

## 9. 커밋 메시지 규칙

1. 커밋 메시지는 항상 `[TYPE] 한국어 설명` 형식으로 쓴다.
2. `TYPE`은 작업 성격을 나타내는 대문자 영문으로 적는다.
3. 메시지는 짧고 구체적으로 작성한다.
4. 예시는 아래와 같다.
   - `[FIX] 검색 필터 검증 추가`
   - `[FEAT] 작업 순서도 문서 추가`
   - `[CHORE] 배포 설정 정리`
